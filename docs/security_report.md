# Security Risk Assessment Report

## 1. 概要 (Overview)
本レポートは、`01_AI_dev` プロジェクト（バックエンド：Go / Echo、フロントエンド）のソースコードおよび構成のセキュリティリスク調査結果をまとめたものです。
静的コード解析およびアーキテクチャのレビューに基づき、現在のセキュリティ状態と潜在的なリスク、およびその対策を記載します。

## 2. 調査結果サマリ (Executive Summary)
全体として、最新のフレームワーク（Echo, GORM, Firebase Auth）を適切に利用しており、**SQLインジェクションやCSRFなどの一般的な重大脆弱性に対する基本的な防御は実装されています。**
ただし、**本番環境へのデプロイにおいて致命的となる設定不備（ハードコードされたシークレットなど）**が存在するため、これらはデプロイ前に確実に修正する必要があります。

## 3. 詳細なリスク評価 (Detailed Risk Assessment)

### 3.1. 認証・認可とシークレット管理 (Authentication & Secrets) - 【重要/High Risk】
*   **リスク**: `cmd/server/main.go` において、`JWT_SECRET` 環境変数が未設定の場合のフォールバック値が `dev-secret-do-not-use-in-prod-xxxxxxxxxxxxxxxx` にハードコードされています。
*   **影響**: 本番環境でこの環境変数の設定を忘れた場合、悪意のある攻撃者がこのシークレットを使用して任意のユーザーのJWTトークンを偽造し、システム全体を乗っ取ることが可能になります。
*   **対策**: 本番環境では必ず強固なランダム文字列を `JWT_SECRET` 環境変数として設定してください。また、コード内のハードコード値は開発環境専用であることを徹底するか、環境変数が未設定の場合はサーバーの起動をパニック（エラー終了）させる設計への変更を推奨します。

### 3.2. SQLインジェクション (SQL Injection) - 【低/Low Risk】
*   **リスク**: データベースとのやり取りにおいて、不正なSQLが実行されるリスク。
*   **現状と評価**: `repository` 層のコードを調査した結果、すべてのクエリで GORM のプレースホルダー（`Where("id = ?", id)` など）が正しく使用されています。文字列連結によるクエリ構築は見られず、**SQLインジェクション攻撃に対して安全に実装されています。**

### 3.3. クロスサイトスクリプティング (XSS) - 【低/Low Risk】
*   **リスク**: 悪意のあるスクリプトがユーザーのブラウザ上で実行されるリスク。
*   **現状と評価**: バックエンドは JSON API として構築されており、`Content-Type: application/json` を返却するため、バックエンド起因の直接的なXSSリスクは低いです。フロントエンド（React/Vite想定）の標準的なエスケープ処理に依存しているため、フロントエンドで `dangerouslySetInnerHTML` などを不適切に使用しない限り安全です。

### 3.4. クロスサイトリクエストフォージェリ (CSRF) - 【低/Low Risk】
*   **リスク**: ユーザーの意図しないリクエストが勝手に送信されるリスク。
*   **現状と評価**: 認証には `Authorization: Bearer <token>` を用いた JWT トークンベースの認証が採用されています。Cookieベースの認証ではないため、**標準的なCSRF攻撃に対しては耐性があります。**

### 3.5. CORS (Cross-Origin Resource Sharing) の設定 - 【中/Medium Risk】
*   **リスク**: 許可されていないドメインから API にアクセスされるリスク。
*   **現状と評価**: `main.go` にて `AllowOrigins: []string{"http://localhost:5173", "http://127.0.0.1:5173"}` と厳格に設定されています。これはローカル開発用としては非常に安全です。
*   **対策**: 本番環境にデプロイする際は、環境変数から許可するオリジン（フロントエンドの本番ドメイン）を読み込むように変更する必要があります。誤って `*` (ワイルドカード) を設定しないように注意してください。

### 3.6. サービス拒否攻撃 (DoS) と レートリミット - 【中/Medium Risk】
*   **リスク**: 大量のリクエストによるサーバーのリソース枯渇。
*   **現状と評価**: `POST` や `DELETE` などのデータ更新を伴うエンドポイントに対して `middleware.RateLimit(100)` (1分間に100回) が適用されています。これにより、ブルートフォース攻撃や簡易的なスパム投稿は緩和されます。
*   **対策**: データの取得（`GET` リクエスト）にはレートリミットがかかっていないため、悪意のあるスクレイピングの標的になる可能性があります。必要に応じて、全体的なAPIレートリミットの導入を検討してください。

### 3.7. 依存関係の脆弱性 (Dependency Vulnerabilities) - 【要確認/Needs Verification】
*   **フロントエンド**: 過去の `npm audit` 実行結果で脆弱性0件（クリーン）であることが確認されています。
*   **バックエンド**: 現在、`go.mod` ファイルから依存関係の記載が欠落している構成の不具合が確認されました。そのため `govulncheck` などの静的解析ツールが正常に機能しません。
*   **対策**: バックエンドの `go mod tidy` を正常に実行できる状態に修復し、定期的に `govulncheck` を CI/CD パイプラインに組み込んでパッケージの脆弱性を監視することを推奨します。

### 3.8. Firebase 認証とパスワード管理 - 【安全/Safe】
*   **現状**: ユーザーのパスワードはローカルのデータベースには一切保存されておらず、認証の複雑な処理は Firebase Authentication に委譲されています。これは認証情報を保護するための**ベストプラクティスに準拠**しています。

## 4. 推奨されるネクストアクション
1.  **環境変数の整備**: 本番環境用の `JWT_SECRET`, `DATABASE_URL`, `FIREBASE_PROJECT_ID`, `CORS_ORIGIN` 等の環境変数管理フロー（Secret Manager 等の利用）を構築する。
2.  **Goモジュールの修復**: `backend/go.mod` の依存関係（requireブロック）が消失している問題を修正し、ビルド可能な状態を担保する。
3.  **セキュリティヘッダーの追加**: Echo のセキュアミドルウェア (`middleware.Secure()`) を追加し、HSTS や X-Frame-Options 等のHTTPセキュリティヘッダーを有効にする。
